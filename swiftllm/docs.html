SwiftLLM Docs

Everything you need to install, configure, and run SwiftLLM for LLM inference, serving, and training — on Linux, macOS, x86_64, or ARM64.

Installation

SwiftLLM provides a smart installer that auto-detects your system, installs dependencies, and builds the engine from source. As of v2.0.1-beta it ships pre-built wheels for Linux (x86_64, aarch64) and macOS (x86_64, arm64).

Prerequisites

  • Python 3.8+ (3.10+ recommended)
  • Git and a C toolchain (cc/clang)
  • NVIDIA GPU with CUDA 11.8+ — optional; pass --cpu for CPU-only builds on any platform
  • Rust toolchain — installed automatically if missing

Standard Install

Shell
# Clone the repository git clone https://github.com/infrabrew/swiftllm.git cd swiftllm # Run the installer (auto-detects CUDA vs CPU) bash install.sh

The installer detects NVIDIA GPU and CUDA toolkit, creates a Python virtual environment, installs Rust via rustup, builds the native extension with maturin, and sets up the model cache.

Install Options

FlagDescription
--cpuForce CPU-only build, skip CUDA detection (works on ARM, Apple Silicon, and older x86_64)
--gpuForce GPU/CUDA build, fail fast if CUDA is missing
--venv DIRCustom virtual environment location (default: ./venv)
--no-venvInstall into the current Python environment
--model-dir PATHSet custom model cache directory
--airgapInstall from a pre-built offline bundle (see Air-Gap section)
--skip-rustSkip Rust installation (assume already present)
--devInstall developer extras (pytest, ruff, maturin in dev mode)

Optional Extras

Install optional dependency groups via pip extras:

Shell
pip install "swiftllm[serve]" # FastAPI/uvicorn server deps pip install "swiftllm[train]" # peft, bitsandbytes, trl pip install "swiftllm[gguf]" # llama-cpp-python for GGUF loading pip install "swiftllm[all]" # everything
⚠

PEP 668 / externally-managed environments: On Debian 12+, Ubuntu 23.04+, and recent macOS systems, the installer will refuse to pip-install into the system Python. Either let install.sh create a venv (the default), pass --venv ~/.swiftllm, or use --no-venv only if you have already activated an environment you control.

Verify the Install

Shell
swiftllm --version swiftllm info --system # prints CPU, GPU, CUDA, wheel target

Quick Start

Basic Inference

Python
from swiftllm import LLM, SamplingParams # Initialize the model llm = LLM("meta-llama/Llama-3-8B") # Set sampling parameters params = SamplingParams( temperature=0.7, top_p=0.9, max_tokens=256 ) # Generate text outputs = llm.generate(["What is machine learning?"], params) for output in outputs: print(output.text)

Interactive Chat

Shell
# Start interactive chat mode swiftllm chat --model meta-llama/Llama-3-8B

Start API Server

Shell
# Launch OpenAI-compatible server swiftllm serve --model meta-llama/Llama-3-8B --port 8000 # Use with any OpenAI client curl http://localhost:8000/v1/completions \ -H "Content-Type: application/json" \ -d '{"model": "llama-3-8b", "prompt": "Hello!", "max_tokens": 64}'

Air-Gapped Installation

SwiftLLM supports deployment on isolated networks with no internet access. Create a portable bundle on a connected machine, then install offline.

Step 1: Create the Bundle

Shell
# On a machine with internet access bash airgap-bundle.sh # Optionally include a model bash airgap-bundle.sh --model TheBloke/Llama-2-7B-GGUF:llama-2-7b.Q4_K_M.gguf

Step 2: Transfer & Install

Shell
# Transfer the archive to the target machine scp swiftllm-airgap-*.tar.gz target-machine:~/ # On the target machine tar xzf swiftllm-airgap-*.tar.gz cd swiftllm-airgap-*/ bash install.sh --airgap
ℹ

The bundle includes the source tree, Python wheels, Rust installer, and optionally pre-downloaded models. SHA256 checksums verify integrity.

Inference

SwiftLLM provides both synchronous and asynchronous inference APIs for generating text from large language models.

Synchronous Generation

Python
from swiftllm import LLM, SamplingParams llm = LLM("meta-llama/Llama-3-8B") params = SamplingParams(temperature=0.8, max_tokens=512) # Batch generation prompts = ["Explain gravity", "Write a haiku"] results = llm.generate(prompts, params)

Async Generation

Python
from swiftllm import AsyncLLM, SamplingParams llm = AsyncLLM("meta-llama/Llama-3-8B") params = SamplingParams(temperature=0.7) result = await llm.generate(["Hello!"], params)

Sampling Parameters

ParameterDefaultDescription
temperature1.0Controls randomness. Lower = more deterministic
top_p1.0Nucleus sampling threshold
top_k-1Top-K sampling (-1 = disabled)
max_tokens256Maximum tokens to generate

API Server

SwiftLLM includes an OpenAI-compatible HTTP server built with Axum, supporting streaming responses, API key authentication, and security hardening.

Shell
# Start the server swiftllm serve --model meta-llama/Llama-3-8B --port 8000 --host 0.0.0.0

Endpoints

  • POST /v1/completions — Text completion
  • POST /v1/chat/completions — Chat completion
  • GET /v1/models — List loaded models
  • GET /health — Health check
💡

The server is a drop-in replacement for the OpenAI API. Any client library that works with OpenAI will work with SwiftLLM.

Training & Fine-Tuning

SwiftLLM supports full training, LoRA, and QLoRA fine-tuning with multiple optimizer options. The CLI exposes two subcommands: swiftllm train (full or method-flagged) and swiftllm finetune (LoRA convenience).

⚠

Simulated backend notice (v2.0.1-beta): the CUDA training kernels are not yet wired up. Every training run prints a visible [SIMULATED] banner and executes a fixed 100-step loop with synthetic loss curves. Config, logging, and checkpoint plumbing are exercised end-to-end; no gradient updates are performed. Real gradient training ships in a later beta.

LoRA Fine-Tuning (Python)

Python
from swiftllm import fine_tune, LoRAConfig, TrainingConfig lora = LoRAConfig(r=16, alpha=32, dropout=0.05) config = TrainingConfig( learning_rate=2e-4, num_epochs=3, batch_size=4, optimizer="adamw", mixed_precision="bf16", save_steps=500, ) fine_tune( model="meta-llama/Llama-3-8B", dataset="./my_data.jsonl", lora_config=lora, training_config=config, output_dir="./output", )

CLI Fine-Tuning

Shell
# LoRA fine-tune via the convenience command swiftllm finetune \ --model meta-llama/Llama-3-8B \ --train-data ./my_data.jsonl \ --eval-data ./eval.jsonl \ --output-dir ./output \ --lora-r 16 --lora-alpha 32 \ --learning-rate 2e-4 \ --num-epochs 3 \ --batch-size 4 # Full training (or QLoRA / LoRA via --method) swiftllm train \ --model meta-llama/Llama-3-8B \ --method qlora \ --train-data ./my_data.jsonl \ --output-dir ./output \ --optimizer muon \ --mixed-precision bf16 \ --lr-scheduler cosine \ --save-steps 500

Training Arguments

FlagDescription
--methodfull, lora, or qlora (default: lora for finetune)
--train-dataPath to a JSONL/CSV/text file. Validated up-front — missing, non-regular, or empty files fail fast.
--eval-dataOptional eval dataset, same validation rules.
--configJSON config file; train_data/eval_data inside are validated the same way.
--lora-r, --lora-alpha, --lora-dropoutLoRA rank, alpha, and dropout.
--optimizeradamw, muon, or sgd
--lr-schedulercosine, linear, constant, or warmup
--mixed-precisionfp32, fp16, or bf16
--save-stepsCheckpoint every N steps (default: end of run only)
--gradient-accumulationMicro-batches per optimizer step

Path Validation (new in v2.0.1-beta)

Before v2.0.1-beta, a bogus --train-data path would silently "succeed" — the simulated loop ran anyway and wrote a final checkpoint. That foot-gun is fixed: both cmd_train and cmd_finetune now call _validate_train_data() which raises:

  • FileNotFoundError — missing path or non-regular file (e.g. a directory)
  • ValueError — zero-byte / empty file

The validator runs before any model loading or checkpoint creation, so failed runs produce no side effects.

Optimizers

  • AdamW — Standard adaptive optimizer with decoupled weight decay. Safe default.
  • Muon — Momentum-based orthogonalized optimizer designed for large transformer layers; pairs well with bf16.
  • SGD — Classic SGD with momentum for full-parameter training on smaller models.

FAQ

Can I run SwiftLLM without a GPU?+

Yes. Pass --cpu to install.sh or set SWIFTLLM_FORCE_CPU=1. Inference and the serving stack run on pure CPU with no CUDA runtime. Tokens/sec will be lower than on GPU, but functionality is identical.

Does SwiftLLM work on ARM / Apple Silicon?+

Yes. v2.0.1-beta ships wheels for Linux aarch64 and macOS arm64 (and x86_64). These are CPU-only today; a Metal backend for Apple Silicon is on the roadmap. ARM Jetson devices with CUDA work with the standard CUDA build.

Which CUDA versions are supported?+

CUDA 11.8 or later. The installer auto-detects via nvcc; you can override with SWIFTLLM_BUILD_TARGET=cuda or pass --gpu to fail fast if CUDA is missing.

pip says "externally-managed-environment" — what do I do?+

That's PEP 668 kicking in on Debian 12+, Ubuntu 23.04+, and recent macOS. SwiftLLM's install.sh creates a venv by default to avoid this. If you're running pip install manually, either activate a venv first, use pipx, or pass --break-system-packages at your own risk. Don't do the last one on production systems.

Is training really working? I see a [SIMULATED] banner.+

Honest answer: the CUDA training backend is not wired up in v2.0.1-beta. Training runs the config/logging/checkpoint plumbing end-to-end but executes a fixed 100-step synthetic loss curve instead of real gradient updates. That's why the banner prints at the top of every run. Real training lands in a later beta; we chose to be loud about it rather than produce misleading metrics.

My --train-data path is wrong and training used to "succeed" anyway. Is that fixed?+

Yes, fixed in v2.0.1-beta. Both swiftllm train and swiftllm finetune validate the path before doing anything else. Missing files raise FileNotFoundError, empty files raise ValueError, and directories or sockets are rejected. Same checks apply to eval_data and to paths pulled from --config JSON.

How do I use SwiftLLM as a drop-in OpenAI replacement?+

Start the server with swiftllm serve --model <id> --port 8000, then set your OpenAI client's base_url to http://localhost:8000/v1. All standard endpoints work: /v1/completions, /v1/chat/completions, /v1/models, plus /health and /metrics.

Can I deploy to air-gapped environments?+

Yes. Run bash airgap-bundle.sh on a connected machine (optionally with --model to bake in weights). Transfer the resulting .tar.gz to the target, extract, and run bash install.sh --airgap. Set SWIFTLLM_OFFLINE=1 at runtime to forbid any network calls.

Where do I report bugs or ask for help?+

File an issue at github.com/infrabrew/swiftllm/issues. Include the output of swiftllm info --system, your install flags, and the full traceback.
